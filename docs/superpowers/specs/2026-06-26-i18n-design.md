# i18n — Internationalization Design

**Date:** 2026-06-26  
**Scope:** Add Spanish/English support with a language switcher. Architecture must scale to 7 languages and 10+ feature pages.

---

## Goals

- All user-facing strings (UI labels, error messages, status text) are translatable.
- Default locale is inferred from the browser/OS; falls back to Spanish.
- User's locale choice persists across sessions via localStorage.
- A language switcher in the header lets users pick their preferred language, showing a flag and language name.

## i18next Configuration (`src/i18n/index.ts`)

```ts
i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    fallbackLng: 'es',
    defaultNS: 'common',
    ns: ['common', 'csvImport', 'allocation', 'report'],
    detection: {
      order: ['localStorage', 'navigator'],
      caches: ['localStorage'],        // persists user's choice automatically
    },
    resources: { en: { ... }, es: { ... } },
    interpolation: { escapeValue: false },
  })
```

Translation files are bundled as static imports (no HTTP backend needed — this is a Vite client-only app).

`src/main.tsx` imports `./i18n/index` before mounting the React tree.

## Locale Detection & Persistence

Detection order: `localStorage` → `navigator` (browser/OS language) → fallback `'es'`.

- **First visit:** `navigator.language` is read. If it matches `'en'` or `'es'`, that locale is used; otherwise falls back to `'es'`.
- **Returning visit:** The value stored in `localStorage` under key `i18nextLng` is used.
- **User switches language:** `i18next.changeLanguage(code)` is called from `LanguageSwitcher`. The detector writes the new value to `localStorage` automatically — no custom persistence code required.

## Store Error Handling

The Zustand store cannot use the `useTranslation` hook (not a React component). Instead of building translated strings inline, the store saves a structured error descriptor.

`TranslatableError` is defined in `src/shared/types.ts` alongside the other shared types:

```ts
export interface TranslatableError {
  key: string
  params?: Record<string, string | number>
}
```

The `assignError` field in `WorkspaceStore` changes from `string | null` to `TranslatableError | null`. Existing tests in `useWorkspaceStore.test.ts` that assert `assignError` as a string must be updated to assert the structured object instead.

Example store code (before vs after):

```ts
// Before
set({ assignError: `Assignment Blocked: "${participant.name}" requested...` })

// After
set({
  assignError: {
    key: 'errors.assignmentBlocked',
    params: {
      name: participant.name,
      requestedRoom: participant.requestedRoomType.join(' / '),
      requestedBed: getBedTypeLabel(participant.requestedBedType),
      actualRoom: room.category,
      actualBed: bed.label ?? getBedTypeLabel(bed.type),
    },
  },
})
```

The component rendering the error calls `t(assignError.key, assignError.params)`. The `allocation.json` translation file holds the full message template using i18next interpolation syntax (`{{name}}`).

## LanguageSwitcher Component

**Location:** Header right column, adjacent to the stats widget. Visible on all screen sizes (the stats widget hides on mobile; the switcher remains).

**UI:**
- Button showing current locale: `🇪🇸 Español` or `🇬🇧 English`
- On click: small dropdown listing all supported locales, each with flag emoji + language name
- Active locale is highlighted; clicking another calls `i18next.changeLanguage(code)` and closes the dropdown

**Locale definitions (extensible list):**

```ts
const LOCALES = [
  { code: 'es', flag: '🇪🇸', name: 'Español' },
  { code: 'en', flag: '🇬🇧', name: 'English' },
]
```

Language names are always shown in their own language (e.g. "Español", not "Spanish") — avoids the chicken-and-egg problem of needing to know the UI language to find the right option.

Flags use emoji (no image assets) — renders natively on all target platforms.

## Namespace Ownership

| Namespace | Owns strings from |
|---|---|
| `common` | `App.tsx` (nav labels, stats labels, app title) |
| `csvImport` | `CsvImport.tsx`, `csvParser.ts` error messages |
| `allocation` | `AllocationBoard.tsx`, `ParticipantPool.tsx`, `RoomCard.tsx`, store assignment/auto-allocate errors |
| `report` | `PrintReport.tsx` |

When a new feature page is added, it gets its own namespace file. Existing namespaces are not modified.

---

## Testing

All tests follow behavior-only conventions (no look-and-feel assertions).

- **`LanguageSwitcher.test.tsx`** — renders current locale flag+name; clicking an option calls `i18next.changeLanguage`; dropdown opens/closes correctly.
- **`src/i18n/translations.test.ts`** — imports all namespace JSON files for each language and asserts every key present in `en` also exists in `es` (and vice versa). Prevents missing translations from shipping silently.
- Existing component tests continue to pass — `src/testSetup.ts` is extended with a global i18next initialisation using English translations so all tests get a working `t()` function without per-file boilerplate.

---

## Out of Scope

- Pluralization rules beyond what i18next provides by default.
- RTL layout support.
- Dynamic/server-loaded translation files.
- Translating user-provided data (CSV content, participant names, room names).
